Quando eseguire l'upsize di un database di Microsoft Access in Microsoft SQL Server

Con il passare del tempo quasi tutte le applicazioni di database diventano più grandi e complesse e richiedono il supporto per più utenti. È opportuno considerare pertanto la possibilità di eseguire l'upsize del database di Microsoft Access in Microsoft SQL Server per ottimizzare le prestazioni, la scalabilità, la protezione, l'affidabilità, il recupero e la disponibilità di database e applicazioni. In questo argomento verranno fornite informazioni di riferimento sui motivi principali per l'esecuzione dell'upsize in un database di SQL Server.

Scalabilità e prestazioni elevate

In molti casi Microsoft SQL Server offre prestazioni avanzate rispetto a un database di Access. SQL Server fornisce inoltre il supporto per database di grandi dimensioni, fino a un terabyte, superando di molto il limite corrente per un database di Access, ossia due gigabyte. SQL Server infine funziona in modo ottimale su Microsoft Windows NT elaborando query in parallelo (utilizzando più thread nativi in un unico processo per gestire le richieste degli utenti) e riducendo al minimo i requisiti di memoria aggiuntiva quando si aggiungono altri utenti.

Maggiore disponibilità

L'utilizzo di Microsoft SQL Server consente di eseguire backup dinamici, incrementali o completi, del database in uso. Non è pertanto necessario forzare gli utenti a uscire dal database per eseguire il backup e il database può rimanere in esecuzione 24 ore su 24, sette giorni su sette.

Protezione ottimale

Microsoft SQL Server può integrarsi con la protezione del sistema operativo Windows NT per fornire un accesso singolo alla rete e al database. Ciò facilita notevolmente l'amministrazione di schemi di protezione complessi. Un database di SQL Server su un server è più protetto perché gli utenti non autorizzati non possono accedere al file di database direttamente, ma devono prima accedere al server.

Recupero immediato

Se si verifica un errore di sistema, come un arresto del sistema operativo o un'interruzione dell'alimentazione, Microsoft SQL Server dispone di un meccanismo di recupero automatico in grado di recuperare un database allo stato in cui si trovava in pochi minuti, senza alcun intervento dell'amministratore del database. In questo modo le applicazioni importanti possono essere rieseguite in breve tempo.

Dati distribuiti e transazioni affidabili

L'elaborazione delle transazioni è un requisito essenziale per un sistema strutturato per supportare applicazioni importanti, quali le transazioni bancarie e l'immissione di ordini in linea. Microsoft SQL Server supporta le transazioni atomiche con registrazione in modo che tutte le modifiche apportate in una transazione siano confermate o annullate.

La coerenza e il recupero di una transazione di database sono garantiti anche nel caso di un errore di sistema e durante l'esecuzione di aggiornamenti complessi da parte di più utenti. SQL Server considera tutte le modifiche di database all'interno di una transazione come un'unica unità di lavoro. In base all'impostazione predefinita, possono verificarsi solo due condizioni: una transazione intera viene completata in modo sicuro e tutte le modifiche risultanti vengono applicate al database oppure la transazione viene annullata così come tutte le modifiche apportate al database.

L'utilizzo di un protocollo di esecuzione forzata in due fasi consente a SQL Server di supportare anche le transazioni sincronizzate che occupano più di un server, garantendo così il mantenimento dello stato di coerenza per tutti i server sulla rete.

Elaborazione basata sul server

Microsoft SQL Server è stato progettato come database client/server. I dati e gli indici risiedono su un solo server al quale in molti casi accedono più client sulla rete. SQL Server riduce il traffico di rete elaborando le query di database sul server prima di inviare i risultati al client. In tal modo l'applicazione client/server può eseguire le elaborazioni nella posizione migliore, ossia sul server.

L'applicazione può anche utilizzare le stored procedure e i trigger per centralizzare e condividere la logica applicativa, le regole e le strategie aziendali, le query complesse, nonché il codice di convalida dei dati e di integrità referenziale sul server, piuttosto che sul client.
