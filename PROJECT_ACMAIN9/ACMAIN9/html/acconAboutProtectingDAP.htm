Protezione del collegamento di una pagina di accesso ai dati e del file HTML corrispondente

Una pagina di accesso ai dati è la combinazione di un collegamento memorizzato nel file del database di Microsoft Access (mdb) o del progetto di Microsoft Access (adp) e del file HTML corrispondente che si trova nel file system del computer. Per impedire che il collegamento di una pagina e il file HTML corrispondente vengano rinominati, eliminati o modificati, utilizzare la protezione del file system del computer in cui sono memorizzati i file. Sul computer ad esempio è possibile impostare per la sola lettura il file del database o del progetto di Access contenente il collegamento della pagina. Sul server Web in cui è memorizzato il file HTML corrispondente alla pagina (dopo la pubblicazione della pagina), è possibile impostare per la sola lettura il file e la cartella che lo contiene. Per ulteriori informazioni sulla protezione di file e cartelle, vedere la Guida di Microsoft Windows.

Protezione e controllo dell'accesso al database da una pagina di accesso ai dati

Per garantire un accesso protetto al database di Access o di Microsoft SQL Server al quale è connessa la pagina, considerare quanto segue:

Protezione del database   È possibile connettere una pagina a un database di Access o di SQL Server. Per proteggere l'accesso al database tramite la pagina, è necessario fornire un'adeguata protezione al database utilizzato. Come per qualsiasi applicazione, è necessario impedire l'accesso di utenti non autorizzati al database e controllare il livello di accesso degli utenti collegati.

  • Per un database di Access protetto a livello utente per il quale si desidera attivare questo tipo di protezione mediante una pagina, è necessario accertarsi che le informazioni di connessione per la pagina specifichino il file di informazioni sul gruppo di lavoro corretto e che il file si trovi su una rete pubblica condivisa accessibile a tutti gli utenti. Modificare la connessione alla pagina nella finestra di dialogo Proprietà di Data Link facendo clic sulla scheda Tutti, facendo doppio clic su Jet OLEDB:System database e specificando il percorso UNC del file di informazioni sul gruppo di lavoro appropriato nella casella Valore proprietà. Per ulteriori informazioni, vedere l'argomento relativo alla protezione di un database di Access.

    Se è stata impostata una password di database su un database di Access connesso a una pagina, verrà richiesta l'immissione della password la prima volta che si visualizza la pagina.

  • Per un database di SQL Server, se si desidera utilizzare la protezione integrata di Windows NT per la connessione da una pagina, è necessario modificare la connessione alla pagina nella finestra di dialogo Proprietà di Data Link facendo clic sulla scheda Connessione e selezionando il pulsante di opzione Utilizza protezione integrata di Windows NT. Per ulteriori informazioni, vedere l'argomento relativo alla protezione di un database di SQL Server.

Non salvare il nome utente e la password con la pagina   Se si salvano il nome utente e la password del database con la pagina, qualsiasi utente può connettersi al database. Affinché la pagina richieda l'immissione della password per l'accesso al database, modificare la connessione alla pagina nella finestra di dialogo Proprietà di Data Link facendo clic sulla scheda Connessione e deselezionando la casella di controllo Consenti salvataggio password.

Restituzione dei record in base al nome utente   Può essere utile creare una pagina che non richieda l'immissione di un nome utente e di una password ma che rilevi automaticamente l'identità dell'utente e visualizzi i record in base a questa informazione. È possibile ad esempio creare una pagina che consenta agli utenti di modificare il loro file nell'ambito dei dati sul personale. I passaggi necessari per questa operazione sono i seguenti:

  • Creare una pagina in un progetto di Access connesso a un database di SQL Server con protezione integrata di Microsoft Windows NT.

  • Aggiungere una colonna alla tabella dei dipendenti con il nome utente di accesso di Windows NT di ciascun dipendente.

  • Creare una visualizzazione che utilizzi la funzione suser_sname() per rilevare l'utente al momento connesso e impostare tale visualizzazione come origine record della pagina.

Per un esempio dettagliato di questa procedura, consultare il Manuale del programmatore. di Microsoft Office 2000/Visual Basic.

Impedire l'accesso non autorizzato mediante uno script pericoloso

Per la connessione di una pagina alla relativa origine dati viene utilizzato Microsoft Office Data Source Control (MSODSC). Quando un utente apre una pagina, il controllo MSODSC della pagina può utilizzare il nome dell'utente per la connessione al database. Un autore di pagine malintenzionato potrebbe sfruttare questa situazione utilizzando Microsoft Visual Basic Scripting Edition o JScript contro MSODSC per tentare di accedere ai database sui server accessibili all'utente, ma non a lui stesso. La modalità di accesso ai dati utilizzata da MSODSC determina se una pagina può essere considerata sicura a prescindere o se consente l'accesso ai dati tra domini diversi. È possibile configurare MSODSC in modo da utilizzare due modalità di accesso ai dati: a due livelli o a tre livelli.

L'accesso ai dati a due livelli consiste nella connessione diretta all'origine dati secondo la modalità client/server tradizionale: il client (primo livello) stabilisce una connessione diretta al server di database (secondo livello). Per eseguire questo tipo di accesso ai dati, MSODSC crea una connessione diretta all'origine dati tramite un provider OLE DB registrato sul computer dell'utente. Per un database di Access l'accesso ai dati a due livelli utilizza il Provider Microsoft Jet 4.0 OLE DB del computer locale per stabilire la connessione a un file mdb su una condivisione di rete. Per un database di SQL Server, utilizza invece il Provider OLE DB Microsoft for SQL Server del computer locale per stabilire la connessione a un server di database disponibile sulla rete locale.

L'accesso ai dati a tre livelli consiste nell'utilizzo di un terzo componente remoto tra il client e il database. Questo tipo di accesso è generalmente utilizzato per accedere ai dati su Internet. Per l'accesso ai dati a tre livelli con utilizzo di MSODSC il terzo componente è ActiveX Data Objects (ADO) Remote Data Service (RDS), anche denominato Data Factory, eseguito su Microsoft Internet Information Server (IIS).

Poiché l'utilizzo di qualsiasi pagina con accesso ai dati a due livelli viene rilevato da Microsoft Internet Explorer come un tentativo di accesso tra domini diversi, quando un utente apre una pagina con questo tipo di accesso, si verifica una delle tre seguenti situazioni, a seconda delle impostazioni di protezione di Internet Explorer: la pagina viene disattivata automaticamente, viene richiesto se consentire l'accesso ai dati oppure la pagina viene attivata automaticamente.

Le impostazioni di protezione di Internet Explorer definiscono inoltre diverse aree di protezione. Se una pagina si trova su un server Web in un'area sicura, il tentativo di accesso tra domini può essere attivato automaticamente. In un ambiente controllato, quale una rete Intranet aziendale, le pagine funzionano in modo ottimale se si utilizza l'accesso ai dati a due livelli e ci si assicura che vengano pubblicate da un server in un'area di protezione sicura. Questo è il metodo più semplice per proteggersi da tentativi di accesso non autorizzati mediante script pericolosi. Per ulteriori informazioni sulle impostazioni di protezione, vedere la Guida di Internet Explorer.

Una pagina che utilizza l'accesso ai dati a tre livelli per la connessione a un database viene considerata sicura indipendentemente dall'area di protezione di Internet Explorer da cui viene pubblicata. Quando viene aperta, l'utente non riceve avvertimenti su tentativi di accesso ai dati tra domini se non sono stati modificati i valori predefiniti delle impostazioni di autenticazione.

Per informazioni sull'implementazione dell'accesso ai dati a due o tre livelli, vedere il Manuale del programmatore di Microsoft Office 2000/Visual Basic.
