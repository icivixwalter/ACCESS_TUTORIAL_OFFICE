Protezione dei file di database di Microsoft Access

Il metodo di protezione più semplice consiste nell'impostazione di una password per l'apertura del database di Microsoft Access (mdb). Dopo aver impostato tale password, a ogni apertura del database verrà visualizzata una finestra di dialogo con la richiesta della password e l'apertura del database sarà consentita solo agli utenti che digiteranno la password corretta. Questo metodo è sicuro, in quanto la password viene crittografata in modo che non possa essere letta direttamente nel file, ma si applica soltanto all'apertura del database. Una volta aperto il database, tutti gli oggetti in esso contenuti saranno disponibili per l'utente, a meno che non siano stati impostati altri tipi di protezione descritti più avanti. Per un database condiviso da un piccolo gruppo di utenti o su un solo computer, spesso è sufficiente impostare una password.

Attenzione   Non utilizzare una password se si desidera replicare un database. I database replicati non possono essere sincronizzati se sono state impostate le password.

Per proteggere ulteriormente il database è possibile crittografarlo. Con la crittografia il file del database viene compattato e reso indecifrabile dai programmi di utilità e di elaborazione testi. Decrittografando il database si annullano gli effetti della crittografia.

Protezione degli oggetti di database a livello utente

Il metodo più flessibile e diffuso per la protezione dell'accesso a un database è la protezione a livello utente. Questa forma di protezione è simile ai metodi utilizzati nella maggior parte dei sistemi di rete. Le ragioni principali per cui ricorrere alla protezione a livello utente sono due:

  • Impedire agli utenti di alterare involontariamente un'applicazione modificando tabelle, query, maschere, report e macro su cui si basa l'applicazione.

  • Proteggere i dati sensibili nel database.

Se è impostata la protezione a livello utente, gli utenti devono utilizzare un ID di identificazione e digitare una password quando avviano Microsoft Access. Nell'ambito del file di informazioni sul gruppo di lavoro gli utenti sono identificati come membri di un gruppo. In Microsoft Access sono forniti due gruppi predefiniti, quello degli amministratori (denominato gruppo Amministratori ) e quello degli utenti (denominato gruppo Utenti), ma è possibile definire altri gruppi.

L'impostazione della protezione a livello utente è un'attività lunga e complessa per la maggior parte dei database, ma con l'Impostazione guidata Protezione tutto è più facile e veloce grazie a una procedura che consiste in un solo passaggio. Implementando inoltre schemi di protezione comuni, l'Impostazione guidata Protezione riduce al minimo, se non elimina del tutto, la necessità di utilizzare il comando Protezione del menu Strumenti.

Dopo aver eseguito l'Impostazione guidata Protezione, è possibile assegnare o rimuovere le autorizzazioni per gli account utente e gli account di gruppo facenti parte del gruppo di lavoro per un database e le sue tabelle, query, maschere, report e macro esistenti. È inoltre possibile impostare le autorizzazioni predefinite assegnate da Microsoft Access alle nuove tabelle, query, maschere, report o macro create in un database.

Le autorizzazioni vengono assegnate a gruppi e utenti per definire con quali modalità è consentito utilizzare ciascuna tabella, query, maschera, report e macro contenuti in un database. È possibile ad esempio consentire ai membri del gruppo Utenti di visualizzare, immettere o modificare i dati in una tabella Clienti, ma non di modificarne la struttura. È possibile altrimenti consentire al gruppo Utenti soltanto di visualizzare i dati di una tabella che contiene dati relativi agli ordini, ma impedire loro l'accesso alla tabella Pagamenti. I membri del gruppo Amministratori dispongono di autorizzazioni complete per tutte le tabelle, le query, le maschere, i report e le macro del database. È possibile impostare un sistema di controllo più accurato creando degli account di gruppo personalizzati, assegnando a questi gruppi autorizzazioni mirate e aggiungendo degli utenti ai gruppi.

Se sono sufficienti un gruppo Amministratori e un gruppo Utenti per proteggere il proprio database, non sarà necessario creare altri gruppi e si potranno utilizzare quelli predefiniti (Amministratori e Utenti). In questo caso si dovrà semplicemente provvedere ad assegnare le autorizzazioni appropriate al gruppo Utenti predefinito e ad aggiungere eventuali altri amministratori al gruppo Amministratori predefinito. Tutti i nuovi utenti aggiunti verranno inclusi automaticamente nel gruppo Utenti. Le autorizzazioni assegnate al gruppo Utenti in genere comprendono la lettura e l'aggiornamento dati per le tabelle e le query e l'apertura/esecuzione per le maschere e i report.

Se si desidera un controllo più articolato su diversi gruppi di utenti, è possibile creare altri gruppi, assegnare loro diversi set di autorizzazioni e aggiungere gli utenti ai gruppi appropriati. Per una più agevole gestione delle autorizzazioni si consiglia di assegnarle soltanto ai gruppi, non agli utenti, e di includere in seguito gli utenti nei gruppi adeguati.

Nota   Per ulteriori informazioni sul modello di protezione di Microsoft Access, consultare il Manuale del programmatore di Microsoft Office 2000/Visual Basic.

Protezione del codice Visual Basic, Applications Edition (VBA)

È possibile proteggere il codice Microsoft Visual Basic, Applications Edition (VBA) presente nei moduli e nei moduli sottostanti le maschere e i report con una password da immettere una sola volta per ogni sessione. La password evita che utenti non autorizzati possano modificare, tagliare, incollare, copiare, esportare e cancellare il codice VBA.

Per proteggere la proprietà intellettuale del codice, è possibile rimuovere dal database il codice VBA modificabile e impedire la modifica della struttura delle maschere, dei report e dei moduli salvandolo come file MDE.

Per ulteriori informazioni, vedere l'argomento relativo alla protezione del codice VBA.

Protezione delle pagine di accesso ai dati

Per proteggere una pagina di accesso ai dati, oltre che i dati a cui si accede mediante tale pagina, attenersi alla seguente procedura:

  • Proteggere il database di Access contenente il collegamento alla pagina di accesso ai dati e il corrispondente file HTML mediante la protezione del file system del computer in cui sono memorizzati questi file.

  • Proteggere il database collegato alla pagina di accesso ai dati impedendone l'accesso agli utenti non autorizzati, controllando il livello di accesso una volta che l'utente si è connesso e non salvando la password del database con la pagina di accesso ai dati.

  • Impedire l'accesso non autorizzato da script di Visual Basic potenzialmente pericolosi servendosi delle opzioni di protezione di Microsoft Internet Explorer e utilizzando un sistema di accesso ai dati su tre livelli.

Per ulteriori informazioni, vedere l'argomento relativo alla protezione di una pagina di accesso ai dati.

Protezione in ambiente multiutente

In molti casi potrebbe rivelarsi necessario impedire agli utenti di replicare un database. Consentire a un utente di replicare un database significa permettergli di effettuare una copia di un database condiviso e di aggiungere campi o apportare altre modifiche al database corrente. Sarà probabilmente necessario anche impedire agli utenti di impostare una password per l'accesso al database. Se un utente impostasse una password di accesso a un database condiviso, nessun altro utente sarebbe in grado di aprirlo senza immettere tale password. Un'altra limitazione da imporre agli utenti potrebbe essere quella di impedire loro la modifica delle proprietà di avvio che specificano funzionalità quali menu personalizzati, barre degli strumenti personalizzate o la maschera di avvio.

Se per un database condiviso non è definita la protezione a livello utente, non sarà possibile impedire agli utenti di apportare una qualsiasi delle modifiche sopra indicate. Eseguendo l'Impostazione guidata Protezione per un database si definiscono le opzioni di protezione a livello utente, il che consente di controllare l'accesso a determinate funzionalità e definisce il modo in cui si possono utilizzare gli oggetti del database. Quando è impostata la protezione a livello utente, per poter replicare un database, impostare per esso una password o modificare le sue proprietà di avvio, un utente o un gruppo deve disporre dell'autorizzazione di Amministratore. Una volta eseguita l'Impostazione guidata Protezione, soltanto i membri del gruppo Amministratori del gruppo di lavoro corrente disporranno delle autorizzazioni di amministratore.

Se la protezione a livello utente è già definita e un utente o un gruppo dispone dell'autorizzazione di amministratore per un database, rimuovendo tale autorizzazione si impedirà all'utente o al gruppo di apportare una qualsiasi delle modifiche indicate in precedenza. Se si ha necessità di consentire a un utente o a un gruppo di eseguire queste attività, è possibile assegnare all'utente o al gruppo desiderato l'autorizzazione di amministratore. Non è possibile controllare separatamente l'accesso a queste tre attività.
