Suggerimenti per l'ottimizzazione delle prestazioni dei database SQL esterni

Se i dati nell'applicazione esterna/front end consistono esclusivamente in tabelle di Microsoft SQL Server, è possibile utilizzare indifferentemente un progetto di Microsoft Access o un database di Microsoft Access come front end dell'applicazione. Se tuttavia si desidera che i dati dell'applicazione esterna/front end comprendano sia tabelle di Microsoft Access che tabelle di Microsoft SQL Server, è necessario utilizzare come front end un database di Access.

Quando ci si collega ad una tabella di un database SQL esterno, è possibile ottenere le migliori prestazioni effettuando il collegamento alle tabelle SQL invece di aprirle direttamente. È possibile aprire direttamente le tabelle SQL esterne solo con il codice di Visual Basic. Le tabelle collegate sono molto più rapide, potenti ed efficaci delle tabelle aperte direttamente.

Ulteriori suggerimenti per le prestazioni

  • Recuperare solo i dati necessari. Progettare le query in modo da limitare il numero di record recuperati e selezionare solo i campi necessari per consentire il trasferimento di un minor numero di dati sulla rete.

  • Se non si stanno recuperando molti record, non si stanno aggiornando i dati e non è necessario vedere le modifiche apportate da altri utenti, non utilizzare set di risultati aggiornabili (dynaset). Per fare in modo che i set di risultati non vengano aggiornati in una maschera, impostare la proprietà Tipo Recordset della maschera a Snapshot. Gli snapshot sono più rapidi da aprire e da scorrere rispetto ai dynaset.

  • Se è necessario recuperare un gran numero di record, un dynaset risulta più rapido ed efficace rispetto ad uno snapshot. Lo spostamento alla fine di uno snapshot, ad esempio, richiede il caricamento dell'intero set di risultati sul computer locale, mentre con un dynaset viene caricata solo l'ultima schermata di dati. Inoltre, per aggiungere nuovi record ad una tabella, maschera o query in modo rapido, scegliere il comando Immissione dati dal menu Record. Se la proprietà Tipo Recordset è impostata a Snapshot, il comando Immissione dati non è disponibile.

  • Utilizzare la memoria cache. Se gli ultimi dati recuperati dal server vengono riutilizzati durante l'esecuzione dell'applicazione, si consiglia di recuperare un'unica ampia parte di dati (molte righe) e memorizzarla nella cache piuttosto che recuperare molte singole righe, in quanto la prima operazione risulta molto più rapida della seconda. Nelle maschere e nei fogli dati viene utilizzata automaticamente una cache. Se si stanno recuperando dati con un oggetto Recordset creato in Microsoft Visual Basic, è possibile utilizzare la proprietà CacheSize per specificare quanti record per volta vanno recuperati e copiati nella memoria locale.

  • Evitare di utilizzare query che consentono di elaborare i dati su un computer client locale. Durante l'accesso ai dati esterni, il modulo di gestione di database Jet consente di elaborare i dati a livello locale solo quando l'operazione non può essere eseguita dal server di database esterno. Le operazioni delle query eseguite a livello locale, definite dai comandi SQL utilizzati per realizzarle, includono:
    • Operazioni JOIN tra tabelle di diverse origini dati remote. Si noti che se il collegamento con join viene effettuato tra una tabella locale o una query con pochi record ed una tabella remota con un numero maggiore di record, ed il campo di join della tabella remota è indicizzato, vengono restituiti solo i record che corrispondono alla tabella locale o alla query, ottenendo quindi migliori prestazioni delle query.

    • Operazioni JOIN basate su una query con il predicato DISTINCT o una proposizione GROUP BY.

    • Join esterni contenenti una sintassi non supportata dal server.

    • Predicati DISTINCT contenenti le operazioni che non è possibile elaborare a livello remoto.

    • Operatore LIKE utilizzato con i campi Testo o Memo. È possibile che non sia supportato da alcuni server.

    • Formule e argomenti GROUP BY a più livelli, ad esempio quelli utilizzati nei report con più livelli di raggruppamento.

    • Argomenti GROUP BY basati su una query con un predicato DISTINCT o una proposizione GROUP BY.

    • Query a campi incrociati che hanno più aggregazioni o che hanno intestazioni di campi, righe o colonne contenenti aggregazioni, oppure che hanno una proposizione ORDER BY definita dall'utente.

    • Predicati TOP n oppure TOP n PERCENT.

    • Funzioni definite dall'utente, oppure operatori o funzioni che non sono supportati dal server.

    • Combinazioni complesse di operazioni INNER JOIN, LEFT JOIN oppure RIGHT JOIN nelle query nidificate.
  • Per query di aggiornamento collettivo per origini dati ODBC, ottimizzare le prestazioni sul server impostando a Sì la proprietà FailOnError. Per ulteriori informazioni, fare clic su .
