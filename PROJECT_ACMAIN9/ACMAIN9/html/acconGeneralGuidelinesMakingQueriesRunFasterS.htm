Ottimizzazione delle prestazioni delle query

Sono disponibili diverse operazioni per rendere più veloce l'esecuzione delle query. Oltre ai seguenti suggerimenti è possibile utilizzare l'Analizzatore prestazioni per analizzare determinate query nel database. Per ulteriori informazioni sull'utilizzo dell'Analizzatore prestazioni, fare clic su .

  • Compattare il database. Questa operazione è in grado di velocizzare le query riorganizzando i record delle tabelle in modo tale che questi vengano inseriti in pagine di database vicine e ordinate in base alla chiave primaria della tabella. In questo modo vengono migliorate le prestazioni delle ricerche sequenziali effettuate sui record di una tabella. Infatti per recuperare tutti i record viene letto soltanto il minor numero possibile di pagine di database. Dopo aver compattato il database, eseguire ciascuna query per compilarlo mediante le statistiche aggiornate della tabella.

  • Indicizzare i campi utilizzati per impostare i criteri per la query e indicizzare i campi indice su entrambi i lati di un join oppure creare una relazione tra questi campi. In tal caso il modulo di gestione di database Microsoft Jet crea un indice nella chiave esterna, se questo non esiste già, altrimenti utilizza l'indice esistente.

    Nota   Il modulo di gestione di database Microsoft Jet ottimizza automaticamente una query che unisce una tabella di Microsoft Access sull'unità disco rigido a una tabella sul server ODBC se le dimensioni della tabella di Microsoft Access sono ridotte e i campi uniti sono indicizzati. In questo caso Microsoft Access migliora le prestazioni richiedendo al server solo i record necessari. Accertarsi che le tabelle unite aventi origini diverse vengano indicizzate nei campi di join.

  • Per definire un campo in una tabella, scegliere il tipo di dati di dimensioni minori che sia appropriato per i dati inseriti nel campo. Assegnare inoltre ai campi utilizzati in join gli stessi tipi di dati o dati che siano compatibili, ad esempio Contatore e Numerico se la proprietà Dimensione campo è impostata a Intero lungo.

  • Quando si crea una query, aggiungere solo i campi necessari. Nei campi utilizzati per impostare i criteri deselezionare la casella di controllo Mostra se non si desidera che questi vengano visualizzati.

  • Se la proprietà Origine record di una maschera o di un report è impostata a un'istruzione SQL, salvare tale istruzione come query e impostare quindi la proprietà Origine record al nome della query. Per ulteriori informazioni, fare clic su .

  • Evitare di inserire dei campi calcolati nelle sottoquery. Se si aggiunge una query contenente un campo calcolato a un'altra, è possibile che l'espressione presente nel campo calcolato rallenti le prestazioni sulla query principale. Nell'esempio riportato di seguito la query Q1 viene utilizzata come input per la query Q2. 
    Q1: SELECT IIF([MiaColonna]="Sì","Ordine confermato","Ordine non confermato") AS X FROM MiaTabella;
Q2: SELECT * FROM Q1 WHERE X="Ordine confermato";

Poiché l'espressione IIf presente nella query Q1 non può essere ottimizzata, neanche Q2 può esserlo. Se un'espressione non ottimizzabile è nidificata all'interno di una sottoquery, non è possibile ottimizzare l'intera query.

Di seguito è riportato un modo alternativo per formulare la query:

Q1: SELECT * FROM MiaTabella WHERE MiaColonna = "Sì";

Se le espressioni sono necessarie per l'output, provare a collocarle in un controllo presente in una maschera o in un report. È possibile ad esempio modificare la query precedente in una query con parametri che richiede il valore di MiaColonna e basare quindi la maschera o il report su tale query. Nella maschera o nel report è poi possibile aggiungere un controllo calcolato che visualizzi "Benvenuto" o "Arrivederci" a seconda del valore specificato in MiaColonna.

Formulare la query in base alle indicazioni riportate di seguito:

PARAMETERS [Per vedere gli ordini confermati, digitare Sì. Per vedere gli ordini non confermati, digitare No.] Testo;
SELECT *
FROM MiaTabella
WHERE MiaColonna = [Per vedere gli ordini confermati, digitare Sì. Per vedere gli ordini non confermati, digitare No.];

Nel controllo calcolato presente nella maschera o nel report inserire quanto segue:

=IIF([MiaColonna]="Sì","Ordine confermato","Ordine non confermato")

  • Quando si raggruppano i record in base ai valori presenti in un campo di join, specificare Raggruppamento per il campo contenuto nella stessa tabella del campo di cui si sta calcolando il totale, includendo nel calcolo un aggregato. Se ad esempio nel database di esempio Northwind si crea una query che totalizza i valori contenuti nel campo Quantità della tabella Dettagli ordini e li raggruppa per ID ordine, è consigliabile specificare Raggruppamento per il campo ID ordine della suddetta tabella. Se si specifica Raggruppamento nel campo ID ordine della tabella Ordini, Microsoft Access deve prima unire tutti i record ed eseguire quindi l'aggregato anziché eseguire l'aggregato e poi unire solo i campi necessari.

    Per velocizzare le operazioni, utilizzare Raggruppamento nel minor numero di campi possibile. Utilizzare come alternativa la funzione Primo, se appropriato. Per ulteriori informazioni sulla funzione Primo, fare clic su .

    Se una query Totali comprende un join, è consigliabile raggruppare i record in una query e aggiungerla in un'altra che esegua il join. In tal modo vengono migliorate le prestazioni di alcune query.

  • Evitare, quando è possibile, di utilizzare criteri restrittivi per una query nei campi calcolati e non indicizzati.

    Utilizzare delle espressioni criterio ottimizzabili. Per ulteriori informazioni sull'ottimizzazione delle espressioni criterio con la tecnologia Rushmore, fare clic su .

  • Se si utilizzano dei criteri di restrizione per i valori di un campo utilizzato in un join fra tabelle con una relazione uno a molti, verificare se la query viene eseguita più rapidamente con i criteri posti sul lato del join definito "uno" oppure su quello definito "molti". In alcune query si ottengono infatti delle prestazioni migliori aggiungendo dei criteri per il campo sul lato "uno" del join anziché sul lato "molti".

  • Indicizzare i campi utilizzati per l'ordinamento.

  • Se i dati non vengono modificati spesso, utilizzare le query di creazione tabella per creare delle tabelle dai risultati delle query. Utilizzare queste tabelle invece delle query come base per le maschere, i report o altre query e accertarsi che gli indici vengano aggiunti secondo le indicazioni generali riportate di seguito.

  • Evitare di utilizzare le funzioni di aggregazione sui domini, quali la funzione DLookup per accedere ai dati da una tabella non presente nella query. Le funzioni di aggregazione sui domini sono tipiche di Microsoft Access e il modulo di gestione di database Microsoft Jet non è in grado di ottimizzare le query che le utilizzano. Aggiungere invece alla query la tabella alla quale deve accedere la funzione oppure creare una sottoquery.

  • Se si sta creando una query a campi incrociati, utilizzare se possibile le intestazioni di colonna fisse. Per ulteriori informazioni, fare clic su .

  • Utilizzare gli operatori BETWEEN...AND, IN e = nei campi indicizzati.

  • Per le query di aggiornamento collettivo in base a origini dati ODBC, ottimizzare le prestazioni sul server impostando la proprietà Errore a Sì. Per ulteriori informazioni, fare clic su .
