Descrizione di OLE DB e dell'ambiente client/server

Un progetto di Microsoft Access utilizza OLE DB per connettersi e comunicare con un database di SQL Server. L'architettura OLE DB è stata progettata da Microsoft per funzionare in modo ottimale in un ambiente client/server.

Quando si accede ai dati tramite una maschera o un foglio dati, OLE DB recupera un set di record di snapshot aggiornabile dal database di SQL server fino al limite massimo di record e memorizza i dati nella cache sul client effettuando un collegamento al server. Access recupera i dati in modo asincrono, consentendo così l'interazione con l'applicazione durante il caricamento dei dati. Quando si eseguono operazioni di esplorazione, filtraggio, ordinamento, ricerca e aggiornamento dei dati sulla maschera o sul foglio dati, i dati utilizzati sono quelli memorizzati nella cache sul client.

Elaborazione dei dati sul server

Per elaborare i dati sul server, è possibile utilizzare le stored procedure e i trigger. I dati sul server possono essere ordinati prima del recupero.

Le stored procedure sono un insieme precompilato costituito da una o più istruzioni SQL e istruzioni facoltative per il controllo del flusso in grado di accettare i parametri ed elaborate come un'unica unità da Microsoft SQL Server. Le stored procedure risiedono nel database di SQL Server. Poiché vengono compilate sul server al momento della creazione, vengono eseguite più rapidamente delle singole istruzioni SQL. Poiché inoltre vengono eseguite sul server, possono usufruire delle maggiori potenzialità di elaborazione del server. Per ulteriori informazioni, vedere l'argomento relativo all'utilizzo delle stored procedure.

I trigger sono un tipo particolare di stored procedure eseguito automaticamente al momento dell'aggiornamento, dell'inserimento o dell'eliminazione dei dati. L'utilizzo dei trigger consente di applicare le regole e le restrizioni aziendali più complesse di quelle definite dai vincoli di controllo. A differenza di questi, i trigger possono fare riferimento alle colonne di altre tabelle. Un trigger ad esempio può annullare gli aggiornamenti che tentano di applicare uno sconto (memorizzato in una tabella Sconti) ai libri (memorizzati in una tabella Libri) con un prezzo inferiore a 18.000 lire. Per ulteriori informazioni, vedere l'argomento relativo all'utilizzo dei trigger.

Per ordinare i record sul server è necessario utilizzare una stored procedure o un'istruzione SQL memorizzata nella proprietà Origine record di una maschera o di un report eseguito sul server. L'ordinamento dei record sul server consente di usufruire delle maggiori potenzialità di elaborazione del server. Per ulteriori informazioni, vedere l'argomento relativo ai metodi di ordinamento.

Limitazione dei dati recuperati dal server

Si consiglia di limitare il più possibile i dati che l'applicazione recupera dal database di SQL Server. Utilizzare visualizzazioni, stored procedure, filtri sul server e proposizioni WHERE delle istruzioni SQL per visualizzare in una maschera o in un foglio dati esclusivamente i dati necessari. È generalmente consigliabile evitare di strutturare applicazioni che consentano agli utenti di sfogliare il database senza alcuna limitazione.

È opportuno utilizzare il pulsante del limite massimo di record o la proprietà Num. massimo record per limitare i dati da visualizzare in una maschera o in un foglio dati.

È anche possibile utilizzare i filtri su server per limitare i record prima che vengano recuperati dal database sul server e altri filtri per limitare ulteriormente i record sul client. Per ulteriori informazioni, vedere l'argomento relativo alla creazione dei filtri. È possibile definire un filtro nella proprietà Filtro su server di una maschera o di un report. È possibile inoltre ottimizzare le prestazioni degli elenchi di valori nelle finestre Filtro su server in base a maschera e Filtro in base a maschera modificando la proprietà Ricerca con filtro per controllare il numero di valori visualizzati a seconda che vengano recuperati dal server o dal client. Per ulteriori informazioni, vedere l'argomento relativo all'ottimizzazione delle prestazioni dei filtri.

Ottimizzazione del database di Microsoft SQL Server

Vengono fornite di seguito indicazioni generali per l'ottimizzazione delle prestazioni del database di SQL Server. Per informazioni dettagliate su come migliorare le prestazioni dei database di SQL Server, vedere la documentazione di SQL Server.

  • Strutturare le tabelle senza dati ridondanti. Un database strutturato correttamente è un requisito essenziale per recuperare e aggiornare i dati con rapidità.

  • Applicare l'integrità referenziale, che conserva le relazioni definite tra le tabelle quando vengono aggiunte, aggiornate o eliminate righe mediante la definizione dei vincoli per le chiavi primarie ed esterne nelle tabelle correlate.

  • Scegliere un tipo di dati appropriato per i campi. È possibile risparmiare spazio nel database e migliorare le operazioni di join. Quando si definisce un campo, scegliere il tipo di dati o le dimensioni di campo minime appropriate per i dati del campo.

  • Creare indici per i campi da ordinare, per cui eseguire il join o per i quali impostare criteri. La velocità di una query può essere notevolmente aumentata indicizzando i campi su entrambi i lati dei join e i campi utilizzati per impostare i criteri della query. Gli indici però possono rallentare l'inserimento, l'aggiornamento e l'eliminazione dei record.

  • Aggiungere un campo data e ora nelle tabelle per migliorare le prestazioni delle operazioni di eliminazione e aggiornamento dei record, in particolare se la tabella contiene molti campi. SQL Server utilizza un campo data e ora per indicare che un record è stato modificato, ma non quando è stata apportata la modifica, creando un campo con valore univoco e aggiornandolo ogni volta che si aggiorna il record. SQL Server utilizza il valore dei campi data e ora per determinare se un record è stato modificato prima di aggiornarlo. Un campo data e ora offre generalmente prestazioni e affidabilità elevate, soprattutto in un ambiente multiutente. Se non è presente un campo di questo tipo, SQL Server deve controllare tutti i campi del record per determinare se quest'ultimo è stato modificato e questa operazione può rallentare le prestazioni.

  • Compattare periodicamente il database di SQL Server per risparmiare spazio su disco e riorganizzare gli indici e i dati per un accesso più rapido.

Ottimizzazione delle maschere

Quando si struttura una maschera, è generalmente consigliabile determinare i dati e le funzionalità necessarie, senza richiedere tali informazioni al server prima che vengano richieste dall'utente. È possibile ad esempio creare una maschera in modo che non sia necessario recuperare i dati dal server durante il processo di apertura aggiungendo un pulsante di comando alla maschera per caricare i dati sulla maschera stessa.

Visualizzare i campi che impiegano molto tempo per recuperare i dati dal server, come i campi di tipo Text e Image, solo quando è necessario. È possibile utilizzare le seguenti tecniche:

  • Posizionare i campi di testo e immagine su una sottomaschera che utilizza la stessa origine record. Verranno estratti solo i valori di testo e immagine per il record corrente.

  • Visualizzare i campi più importanti su una maschera principale e fornire un pulsante denominato "Ulteriori informazioni" che consente di aprire un'altra maschera contenente altri campi. La seconda maschera avrà come base una query con parametri basati sul campo della chiave primaria della maschera principale.

  • Se l'applicazione richiede numerose immissioni di dati, come nel caso di un sistema di immissione di ordini, è opportuno creare una maschera specifica per l'aggiunta di record con la proprietà Immissione dati impostata a Sì. Quando si apre la maschera per immettere nuovi dati, non viene visualizzato alcun record esistente. Ciò consente di risparmiare tempo dato che non devono essere recuperati tutti i record dal server.

  • Nel caso di una maschera o un report basato su una stored procedure con parametri, utilizzare la proprietà Parametri di input per fornire i valori dei parametri.

Utilizzare le istruzioni SQL UPDATE e DELETE con la proposizione WHERE appropriata per aggiornare o eliminare più record che soddisfano gli stessi criteri. Questa operazione è più efficiente dell'apertura di un set di record e dell'esecuzione di aggiornamenti o eliminazioni di un record per volta.

Ottimizzazione dei report

Invece di stampare un lungo report, è consigliabile suddividere i report in sottoinsiemi di dati di dimensioni inferiori utilizzando visualizzazioni, stored procedure e istruzioni SQL come origine record oppure utilizzando i filtri.

Utilizzare l'anteprima di layout piuttosto che quella di stampa nel corso della strutturazione dei report. In questo modo è possibile verificare rapidamente il layout in quanto viene visualizzato solo un campione dei dati. Al termine delle operazioni eseguite sul layout del report, controllare i dati aprendo il report in anteprima di stampa.

I report in genere vengono creati e distribuiti in base a un programma legato ai cicli aziendali, come i rapporti settimanali sulle scorte di magazzino, i rapporti mensili sulle vendite o i resoconti finanziari trimestrali. Creare e distribuire questi report come snapshot di report in un file batch. Gli utenti possono utilizzare Snapshot Viewer per visualizzare e stampare i report. Per i rapporti aggiornati al minuto, utilizzare le pagine di accesso ai dati con gruppi.
