Protezione del codice Visual Basic, Applications Edition (VBA)

In un progetto e in un database di Microsoft Access è possibile proteggere tutti i moduli, inclusi quelli sottostanti alle maschere e ai report, impostando una password che è necessario immettere per visualizzare o modificare il codice VBA in Visual Basic Editor. Dopo l'impostazione, viene richiesta l'immissione della password una volta per ciascuna sessione. Questa password non è solo necessaria per visualizzare e modificare il codice, ma anche per tagliare, copiare, incollare, esportare ed eliminare qualsiasi modulo. Questo tipo di protezione del codice VBA tuttavia non impedisce di eseguire il codice VBA esistente.

Moduli, maschere, report e protezione a livello utente

In un database di Microsoft Access 2000 i moduli, inclusi quelli sottostanti alle maschere e ai report, non sono più protetti a livello utente, come nelle versioni precedenti. Per maschere e report invece è ancora attivo questo tipo di protezione. Vengono quindi utilizzati due diversi meccanismi di protezione, uno per i moduli e l'altro per maschere e report.

Se si dispone delle autorizzazioni necessarie, è possibile ad esempio aggiungere un controllo a una maschera, ma se il progetto VBA è bloccato per la visualizzazione e protetto con una password che non si conosce, non sarà possibile visualizzare o modificare il modulo corrispondente. Al contrario, è possibile impedire la progettazione di una maschera o di un report non concedendo l'autorizzazione a modificare la struttura dell'oggetto mediante la protezione a livello utente, ma se l'utente conosce la password del codice VBA, potrà accedere al modulo corrispondente.

Questa regola prevede un'eccezione. Anche se si dispone dell'autorizzazione per modificare la struttura di una maschera o di un report, non è possibile eliminare l'oggetto o impostare la proprietà Possiede modulo a No, poiché questa azione elimina il modulo della maschera o del report.
