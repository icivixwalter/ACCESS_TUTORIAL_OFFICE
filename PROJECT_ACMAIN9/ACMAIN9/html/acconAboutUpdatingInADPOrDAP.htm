Informazioni sull'aggiornamento dei dati in un progetto di Microsoft Access o in una pagina di accesso ai dati

Il modo in cui i dati vengono aggiornati in un progetto di Microsoft Access o in una pagina di accesso ai dati è diverso dal modo in cui vengono aggiornati in un database di Microsoft Access. Sia gli utenti che i programmatori devono essere consapevoli di queste differenze. In questo argomento verranno fornite informazioni di riferimento relative a:

Regole generali per l'aggiornamento dei dati

Le regole riportate di seguito vengono applicate quando si aggiornano i dati da un progetto o una pagina di Access connessa a un database di Access o di Microsoft SQL Server:

  • Per aggiornare qualsiasi campo (incluso quello della chiave primaria) in una singola tabella o tabella di base di un join, è necessario avere un campo o una combinazione di campi univoca definita come vincolo della chiave primaria, un vincolo univoco o un qualsiasi indice univoco per quella tabella.

  • È possibile aggiornare i dati in base a un join uno-a-molti in una maschera e in una pagina, ma solo per i campi del lato "molti" della relazione. Quando si apre una visualizzazione o si esegue una stored procedure contenente un join in visualizzazione Fogli dati, la visualizzazione o la stored procedure è di sola lettura. È tuttavia possibile aggiornare i dati da una maschera o da una pagina, anche se presenta una visualizzazione o una stored procedure come origine record, impostando la proprietà UniqueTable della pagina o della maschera (vedere più avanti la relativa spiegazione).

  • Per aggiornare una visualizzazione o una stored procedure, è necessario essere in possesso delle autorizzazioni UPDATE, INSERT e DELETE per la tabella o le tabelle di base a cui fa riferimento la visualizzazione o la procedura, nonché l'autorizzazione SELECT per la visualizzazione o l'autorizzazione EXECUTE per la procedura.

  • Non è possibile aggiornare i dati in una pagina con gruppi, ma solo in una pagina senza gruppi.

  • Non è possibile aggiornare i dati derivati da un self-join, ad esempio tutti gli impiegati e i superiori a cui fanno riferimento, in alcuna circostanza.

  • Nel caso di un database di Access:
    • Non è possibile aggiornare una maschera, un foglio dati o una pagina in cui l'istruzione SQL sottostante dell'origine record contiene una proposizione GROUP BY o COMPUTE.

    • Non è possibile aggiornare i campi basati su espressioni, ad esempio "prezzo * quantità AS prezzo_esteso".

    • Non è possibile aggiornare una maschera, un foglio dati o una pagina basata su un join molti-a-molti.

Utilizzo di snapshot aggiornabili

Access 2000 supporta unicamente l'inserimento, l'eliminazione e l'aggiornamento dei dati nella riga di una tabella univoca (il lato "molti" di un join) in uno snapshot aggiornabile creato da un'operazione di join uno-a-molti in una visualizzazione, una stored procedure con restituzione di righe o un'istruzione SQL. Access 2000 non consente di aggiornare i campi sul lato "uno" del join.

Una maschera o pagina basata su un join uno-a-molti deve definire la proprietà UniqueTable perché il set di record sia aggiornabile. La tabella univoca (chiamata anche tabella "most-many") è il lato "molti" di una relazione uno-a-molti. Una tabella è univoca quando un suo record corrisponde a un record della visualizzazione, della stored procedure con restituzione di righe o dell'istruzione SQL. In una visualizzazione che unisce le tabelle Clienti e Ordini ad esempio, Ordini è la tabella univoca perché un record di questa tabella corrisponde, in gran parte, a un record univoco della visualizzazione. Al contrario, un record della tabella Clienti può corrispondere a più record della visualizzazione.

Inoltre, nel caso di un'origine record data da una stored procedure o istruzione SQL contenente marcatori di parametri, se si desidera visualizzare i valori correnti di un record dopo l'aggiornamento, è anche necessario impostare la proprietà ResyncCommand. Se non si imposta la proprietà ResyncCommand in questi casi, l'unica conseguenza è che i valori correnti di un record non saranno visibili dopo un aggiornamento o un inserimento. L'aggiornamento o l'inserimento comunque avrà luogo correttamente. Per altri tipi di origini record vengono visualizzati i valori correnti anche se la proprietà ResyncCommand non è stata impostata.

In uno snapshot aggiornabile sono visibili solo i valori correnti di un record appena aggiunto o modificato dopo che è stato salvato (vedere più avanti la relativa spiegazione). La tabella Ordini ad esempio contiene una colonna ID come chiave primaria. Quando si esegue un inserimento in questa tabella, il nuovo valore IDOrdine verrà visualizzato dopo avere salvato il record o dopo essersi spostati su un altro record (con implicito salvataggio). Analogamente, in una maschera basata su una visualizzazione che unisce Clienti e Ordini, se si aggiorna il campo IDOrdine a un IDCliente diverso, il campo Indirizzo non mostrerà l'indirizzo del nuovo cliente fino a quando il record non sarà stato salvato.

Infine, tutti i campi chiave o la tabella univoca devono trovarsi nell'elenco di selezione delle istruzioni SQL, anche se non necessariamente in forma visibile. È anche necessario selezionare tutte le colonne NOT NULL per cui non sono stati definiti valori predefiniti per il server e fornire valori per queste colonne nel set di record, a eccezione dei campi con la proprietà Identity impostata.

Visualizzazione dei valori dei campi aggiornati

In generale, quando si aggiornano dei campi di una maschera, di un foglio dati o di una pagina, Access è in grado di inserire automaticamente un valore, ad esempio un valore predefinito, un campo calcolato, un valore di data/ora, un campo ID/Contatore o un'azione eseguita da un trigger. Contrariamente a un database di Access tuttavia, la visualizzazione del valore o le conseguenze dell'operazione hanno luogo dopo la conferma del record. Ciò si verifica perché Access deve aggiornare o eseguire l'operazione sul server, quindi aggiornare la visualizzazione corrente sul client.

Metodi consigliati per aggiornare i dati correlati

Di seguito sono elencati i metodi consigliati per aggiornare i dati correlati:

  • Maschera standard costituisce un modo pratico per creare una maschera che consenta l'aggiornamento di una visualizzazione o stored procedure basata su un join. Utilizzare Maschera standard per creare rapidamente la maschera, quindi passare in visualizzazione Struttura per impostare la proprietà Tabella univoca. È anche possibile impostare la proprietà Visualizzazione predefinita se si preferisce una visualizzazione Foglio dati a una visualizzazione Maschera singola.

  • Per aggiornare tutti i dati in una relazione uno-a-molti, è possibile creare una combinazione maschera/sottomaschera. Anziché visualizzare ad esempio i dati di una singola maschera in cui vengono uniti gli ordini e i clienti (in questo caso, i campi cliente sarebbero di sola lettura), è possibile creare una maschera con i clienti e una sottomaschera con gli ordini. Poiché la combinazione maschera/sottomaschera utilizza due set di record diversi, è possibile aggiornare tutti i campi. È inoltre possibile visualizzare questa combinazione maschera/sottomaschera come foglio dati con un foglio dati secondario.

  • In una pagina è possibile simulare una combinazione maschera/sottomaschera collegando due pagine mediante un campo di tipo Collegamento ipertestuale, quindi utilizzare Microsoft FrontPage per creare due frame, uno per ogni pagina.
